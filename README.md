# Buildathon

Buildathon is a Slack-first engineering assistant for Jira, Confluence, Grafana, and local k6 performance workflows.
It can:

- answer grounded questions from enterprise sources
- run Jira-driven performance workflows
- generate ticket-scoped k6 scripts
- execute k6 locally and summarize results
- post Jira comments and Slack updates

## Repository Layout

```text
.
|-- src/                     Python app: agent, connectors, Slack entrypoint
|-- tests/                   Unit tests
|-- performance/             Local k6 workspace, data, scripts, results
|-- docs/skills/             Project skills used by the workflow
|-- website/                 Demo e-commerce stack and observability docs
|-- config.py                Environment-backed settings
`-- requirements.txt         Python dependencies
```

## Main Commands

Examples that work in Slack mentions and in the local CLI:

- `test jira KAN-5`
- `test KAN-5 service=payments vus=2 duration=30s`
- `run k6 test auth vus=2 duration=30s`
- `create k6 report auth`
- `create k6 workflow auth duration=30s`
- `read grafana dashboard payments`
- `create jira ticket summary="Build RAG bot" description="Create the Slack agent flow"`

Natural language requests are also supported. When explicit command parsing does not match, the agent uses intent classification to decide whether the user wants an executable workflow or a search answer.

## Local Setup

Create the virtual environment and install dependencies:

```powershell
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

## Configuration

The app reads `.env` from the repository root.

### Required for Slack

- `SLACK_BOT_TOKEN`
- `SLACK_APP_TOKEN`

### Required for OpenAI-backed workflow generation

- `OPENAI_API_KEY`
- `OPENAI_MODEL`

### Jira

- `JIRA_BASE_URL`
- `JIRA_USERNAME`
- `JIRA_API_TOKEN`
- `JIRA_PROJECT_KEY`

### Confluence

- `CONFLUENCE_BASE_URL`
- `CONFLUENCE_USERNAME`
- `CONFLUENCE_API_TOKEN`
- `CONFLUENCE_SPACE_KEY`

### Grafana

- `GRAFANA_URL`
- `GRAFANA_SERVICE_ACCOUNT_TOKEN`

### k6

- `K6_COMMAND`
- `K6_PROJECT_ROOT`
- `MCP_CONFIG_PATH`

## Running the Bot

Start the Slack Socket Mode app:

```powershell
.venv\Scripts\python.exe -m src.slack_app
```

Run a one-shot local CLI request:

```powershell
.venv\Scripts\python.exe -m src.main "test jira KAN-5"
```

## Running the Demo Website Through Docker

The demo e-commerce platform lives under [website/](website/README.md).
To start the full website, microservices, databases, and observability stack with Docker:

```powershell
cd website
docker compose up -d
docker compose ps
```

Main URLs after startup:

- Frontend: `http://localhost:4000`
- Grafana: `http://localhost:3000`
- Prometheus: `http://localhost:9090`
- Users API: `http://localhost:3001`
- Products API: `http://localhost:3002`
- Cart API: `http://localhost:3003`
- Orders API: `http://localhost:3004`
- Payments API: `http://localhost:3005`

To stop the stack:

```powershell
cd website
docker compose down
```

For a full reset:

```powershell
cd website
docker compose down -v
docker compose up -d
```

## Jira Performance Workflow

The Jira workflow follows this shape:

1. Read the Jira issue.
2. Decide whether the ticket should stop at planning or continue to execution.
3. Build a structured performance plan from Jira text and repository docs.
4. Generate a ticket-scoped k6 script when execution is warranted.
5. Run k6 locally.
6. Generate a report and dashboard artifact.
7. Post a Jira comment and summarize the outcome for Slack.

Generated ticket-scoped scripts are written under:

```text
performance/tests/<service>/<service>.<ticket-key-lower>.test.js
```

These ticket-scoped scripts are runtime artifacts. They can be regenerated by the workflow and are ignored by git by default.

Generated reports are written under:

```text
performance/results/<timestamp>_bot_<service>/
```
